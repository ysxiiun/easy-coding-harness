# 实施 / 审查 / 验证三阶段全程强制子代理（0.3.0）

> 本文档记录 0.3.0 中「强制子代理」一项的修改内容、发布说明与验证方式。
> 0.3.0 同时还包含 README 重写 + 记忆门控修复（另一独立主题），完整版本历史见 [CHANGELOG.md](../../CHANGELOG.md)。

## 背景与目标

运行时 8 阶段状态机中，IMPLEMENT 阶段原先按 `plan` 记录的 `strategy` 字段决定是否开子代理：`single` / `sequential` 都在**主代理里直接写代码**，只有 `parallel` 强制子代理。

问题：子代理不只是为并行，它更关键的作用是**上下文隔离**——把实现细节挡在主代理窗口之外，避免上下文爆炸。`sequential` / `single` 在主代理直接干，正好丢掉了这层隔离。

目标：**实施 / 审查 / 验证三阶段，无论哪种 strategy 都走子代理，哪怕只有一个单元也起一个子代理串行执行，主代理永不亲自写实现代码。**

## 改动前后行为对照

| strategy | 改动前 | 改动后 |
| --- | --- | --- |
| `single` | 主代理直接实现 | 起 **1 个**子代理 |
| `sequential` | 主代理逐个串行实现 | 按 `depends_on` **串行**起子代理（一个 return 再起下一个） |
| `parallel` | 强制子代理 | 按 level **分层并发**起子代理（行为不变） |

`strategy` 字段语义从「是否开子代理」变为「子代理的**编排形态**」——三种都派子代理，区别只在串行 / 并发。

REVIEW / VERIFICATION 本就写了 "ALWAYS USES SUB-AGENTS"，本次只校齐口径、清掉与之矛盾的描述。

## 发布说明

- **版本归属**：随 `0.3.0` 发布——这批改动含强制子代理（运行时行为变更）等功能升级，从 `0.2.x` 进位到 minor。
- **强制方式**：**仅 Prompt 强约束**——只改 skill / agent 的提示词，未改 `easy_coding_state.py` hook，不引入 execution.jsonl 代码级闸门。约束靠 agent 遵守 HARD-GATE，非机械拦截。
- **对存量项目**：已初始化的项目需执行 `easy-coding upgrade` 才能拿到新模板。upgrade 只覆盖 skills / hooks / agents / 主约束文件，**不碰** memory / tasks / spec / sessions。

## 升级到业务项目

同步是两段路，缺一段都吃不到改动：

```
src/templates/   →   templates/              （靠 npm run build / copy-templates）
templates/       →   目标项目 .claude/skills/ （靠 easy-coding upgrade）
```

### 1. 本地构建（在 harness 项目）

```bash
./install.sh          # = npm install + npm run build + npm link
# 或日常只改了模板时,更快:
npm run copy-templates
```

说明：本地 `easy-coding` 命令通过 `npm link` 指向本仓库 `dist/cli.js`，`templates/` 与 `dist/` 平级（模板**不在** dist 里）。link 只需首次建立一次，之后改完 build 即生效。

### 2. 业务项目升级

```bash
cd <你的业务项目>
easy-coding upgrade --dry-run   # 预览将覆盖的文件
easy-coding upgrade -y          # 实际覆盖
```

### 版本判断

upgrade 靠版本号判断要不要升。`0.3.0` 是首个带本批改动的版本，所有存量项目的 `harness_version` 都 < `0.3.0`，因此 `easy-coding upgrade` 必然触发，无需任何手动兜底。

```bash
for p in "implements directly" "implement code directly" \
```


### 业务项目验证（upgrade 后）

确认新版 skill 已落地（Claude 平台）：

```bash

```

端到端：开一个新任务跑到 IMPLEMENT 阶段，观察 `single` / `sequential` 策略下是否也派出了子代理（execution.jsonl 应出现每个单元的 `dispatch` / `result` 记录），主代理不再内联写实现代码。

## 已知限制与后续

- **仅 prompt、非铁闸**：HARD-GATE 是提示词约束，依赖 agent 遵守。措辞已大幅加硬，但理论上仍可能被绕过——尤其 `single` 这种「只有一个任务」的场景，agent 容易自作主张直接做。
- **若实测仍被绕**：再上 hook 层硬强制——在 `easy_coding_state.py` 的 IMPLEMENT→REVIEW 转换处校验 execution.jsonl 是否含子代理 `dispatch` / `result` 留痕，缺则拒绝推进。注意 hook 只能验证「留痕」不能验证「真起了子代理」，但报错能把绕过的 agent 拉回重做。
