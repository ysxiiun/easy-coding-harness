Easy Coding Harness

给 AI 编程套上一套工作流规范——让它有章可循，而不是全凭 agent 当时的发挥。

固定阶段状态机 approve / guard / lite / auto 确认模式 子代理隔离上下文 项目记忆分层 一份模板 · 三平台 Supermodule 分层
来源：docs/introduction.md 版本：0.8.1-beta.2 适配：Claude Code · Codex · Qoder

01它解决什么问题

AI 写代码越来越能干，但放开手让它自己跑，几个老毛病会反复出现。每个痛点，harness 都有对应的机制去堵。

需求还没对齐就开始写，方向错了返工一大片。

→ 项目与 session 分层确认模式

说"改完了"，但 lint、类型、测试根本没真跑过。

→ 验证硬门控

会话一长，上下文塞满实现细节，主线越聊越糊涂。

→ 子代理隔离

换一个 agent 接手，前因后果全断，得从头解释一遍。

→ 交接 / 记忆沉淀

团队里 Claude Code、Codex、Qoder 各用各的，规则各配一套。

→ 一份模板三平台

Easy Coding Harness 不是又一个编程工具，而是给 AI 编程套上的一套工作流规范。它把"什么时候必须停下来让人确认""哪些活儿必须验证过才算数""记忆怎么跨会话留下来"这些纪律，固化成 agent 装好即用的 skills、hooks 和约束文件。

让 AI 编程有章可循，而不是全凭 agent 当时的发挥。

02设计理念

四条世界观，讲的是"为什么这么设计"。

1

纯部署，理解交给运行时

CLI 只往 .claude / .agents / .qoder 铺文件，不读一行项目代码。项目理解留给装好后的 ec-init 现场完成。CLI 升级不碰你的项目知识，边界干净。

2

人机共创，关键节点设硬门控

状态边由 approve、guard（默认）、lite、auto 控制，session 可覆盖项目配置。guard/auto 的代码主链进入 REVIEW，lite 直接进入 VERIFICATION；所有模式都保留 VERIFICATION 与产物门控。

3

上下文卫生

实现、审查、验证都派子代理去干。实现细节是上下文的噪声源，隔离在子代理里，主代理只拿结论，主线才能一直清醒。

4

记忆要沉淀，不能只活在当前会话

任务中写短期记忆，攒够阈值才蒸馏成长期记忆。跨会话、跨仓库还记得住"当初为什么这么改"；量不够就是 no-op，不把琐碎信息固化成经验。

03特色亮点

三个落地武器，讲的是"具体给到什么"。

一条能恢复、能交接的工作流

任务状态持久化在 .easy-coding/ 里，不绑死在某次会话上。中途切换、审查回退、换人接手，都接得回来。

一份模板，喂三个平台

三个平台用同一份 skill 模板，靠安装时替换占位符适配各自的原生约定。不为每个平台各写一套规则，团队里不管谁用哪个 agent，遵循的是同一套纪律。

按 git 边界分层的 Supermodule 支持

在带 .gitmodules 的父仓里，harness 按 git 边界分层运行，父仓和每个子仓各拿一套完整运行时。

边界行为
安装父仓必装，已检出的子仓可选装；不会替你 git submodule update --init。
运行跨仓任务在父仓根跑，用父仓全景记忆；单仓任务进子仓跑，用子仓自己的记忆。
记忆属于某个子仓的技术记忆写回该子仓，单独 clone 也带得走改动原因。
提交跨仓改动两段式提交，先各子仓、再父仓 gitlink。
范围提示：当前支持到一级 submodule，不自动处理子仓里的二级 submodule。

04使用方法

四步跑通，之后每个需求都从 /ec-workflow 进。

1

安装 CLI

0.x 内测阶段推荐 beta 通道，或源码安装。

# npm 内测环境（0.x 阶段推荐）
npm install -g easy-coding-harness@beta

# 或源码安装（开发者 / 内网）
git clone <repo> && cd easy-coding-harness && ./install.sh

easy-coding --version
2

在项目里部署 harness

在目标项目根目录运行。若当前目录是 supermodule 父仓，init 会列出已检出的子仓让你选。

easy-coding init                                  # 默认装 Claude Code
easy-coding init --agent=claude-code,codex,qoder  # 指定多个平台
3

初始化项目知识

打开目标 agent，运行 /ec-init。agent 会读项目，生成 SOUL.md、RULES.md、ABSTRACT.md、TEST_STRATEGY.md 等知识文件。这一步幂等，重复跑安全。

/ec-init
4

日常开发从统一入口进

ec-workflow 负责创建或恢复任务，并按生效确认模式推进状态。需要原生处理当前会话时可调用 ec-no-harness，其他 skills 与 hooks 不受影响。

/ec-workflow 实现 xxx 功能

常用命令

命令用途
easy-coding init首次接入项目，部署所选平台的全部 harness 文件
easy-coding add-agent给已接入项目追加某个平台支持
easy-coding upgradeCLI 升级后同步项目内生成文件，用户资产保留
easy-coding update更新全局 CLI 到最新发布版
easy-coding config交互修改项目级确认模式
easy-coding status查看已安装平台、harness 版本、确认模式与当前任务状态
easy-coding clear移除 harness 安装物，保留 tasks、spec、memory 等用户资产
一句话记住整个流程：装 CLI → init 铺文件 → /ec-init 让 agent 认识项目 → /ec-workflow 开始干活。之后每个需求都从 /ec-workflow 进，剩下的交给状态机。
