## Technical Memory Candidates

- Architecture / Interface Decisions: 新增受限 `auto-transition`，允许 INIT → ANALYSIS、已完成记忆处理后的 MEMORY → COMPLETE，以及通过结果校验的只读 IMPLEMENT → COMPLETE；其他边仍使用 `request-transition` / `confirm-transition`。
- Engineering Rules / Workflows: 代码 IMPLEMENT 完成后提供进入 REVIEW、跳过 REVIEW 进入 VERIFICATION、交接和 free-form Other；代码任务的 VERIFICATION 不可跳过。显式 `doc` / `analysis` / `report` 只读任务只生成 dev-spec 和 execution plan，不生成 `test-strategy.md`；IMPLEMENT 必须留下匹配的 dispatch/result，原样展示完整 `deliverable` 后直接自动 COMPLETE，不进入 REVIEW、VERIFICATION 或 MEMORY，也不写任务记忆。状态 API 要求最新计划为 single + `files:[]`，最新结果为 `changed_files:[]`、非空 deliverable 且无 issues/needs_attention。
- Implementation Patterns / Reusable Approaches: ANALYSIS 固定执行“原样落盘骨架 → 只读识别并即时询问决策 → 决策清零 → 填充完整方案”；最终报告不含 `[阶段：ANALYSIS]` 或“待用户决策”章节。骨架使用 `[[EC_TODO:...]]` 专用标记，避免与方案内合法的 `{title}` 等模板文本冲突。
- Pitfalls / Fix Strategies: 自动接口必须使用固定边白名单，避免变成通用确认绕过；分析中不得先写带假设的半成品方案再让用户统一回答；状态 API 不能只检查标题和 `units` 数量，还要校验必填章节正文、实施单元任务卡字段、依赖图无环与并行分组层级。显式 `doc` / `analysis` / `report` 任务可用 single 空文件范围返回只读 `deliverable`，代码任务和并行计划仍禁止空范围；skills 内置 JSON 示例也必须作为可解析契约接受 unit/group 一致性测试。
- Verification Experience: 状态机升级需同步源模板、构建模板、主约束、文档和集成测试；使用全量 build/typecheck/lint/test、源模板与构建模板 diff、版本扫描、diff check 与 npm pack dry-run 形成闭环。介绍页可由版本测试直接绑定 `package.json`，防止静态 HTML 版本漂移。
