## 技术记忆候选

- 架构 / 接口决策：人工确认边只有在用户可实际选择完整业务分支后才算展示完成；状态 API 继续只负责记录和消费 `pending_transition`，不从用户文本自动迁移状态。
- 工程规则 / 工作流：平台提供原生选择能力时必须在同一轮实际调用；普通确认门提供确认目标阶段和交接两个显式分支，修改意见由原生 free-form Other 承接；无原生能力时才用完整文本编号回退。
- 实现模式 / 复用写法：关键交互不只写在中央 workflow，还要在阶段结束入口和主约束重复最小硬门，并用三平台生成产物测试锁定确认、交接、Other 和失败恢复契约。
- 易错点 / 修复策略：仅写“优先原生选择”或让阶段 skill 引用抽象的 standard gate，容易被 Agent 压缩为单一“回复确认执行”；原生选择空值、取消、超时或无法解析时必须保留待确认边并重新展示完整选项。
- 验证经验：模板协议变更需执行三平台安装回归、全量 typecheck/lint/test/build、源与构建模板一致性、版本一致性、diff check 和 npm pack dry-run。
