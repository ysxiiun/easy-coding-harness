# 0.8.2-beta.0 原生选择超时降级

## 任务摘要

- 目标：避免原生确认选择框超时后必须等下一次交互重新唤起，确保用户稍后可直接回复文本编号继续流程。
- 范围：workflow / analysis skills、Claude/Codex/Qoder 主约束、三平台生成测试、设计与使用文档、版本元数据。
- 结果：已完成；只有原生工具明确保证无限等待时才不预先展示文本降级，否则先输出持久化的完整编号选项，再调用一次原生工具。
- 恢复规则：恢复任务时先根据已保存的 `pending_transition` 消费当前用户回复的编号，再决定是否重新展示门禁；普通门映射 `1/2/3`，Approve 模式代码 IMPLEMENT 特殊门映射 `1/2/3/4`。
- 关键约束：不修改状态机合法边、`pending_transition`、hook 状态 API 或 Claude hook 执行 timeout；本轮未提交、未推送、未发布 npm。

## 技术记忆候选

- 架构 / 接口决策：CLI 仍是纯文件部署器，不能跨平台控制客户端 UI 生命周期；仅当原生工具明确保证无限等待时才单独依赖原生框，否则必须在调用前预先输出文本编号降级。
- 工程规则 / 工作流：原生选择返回空值、取消、超时或无法解析时必须保留 `pending_transition`，不重试原生框；如果持久文本降级已经可见，不重复输出。恢复时必须先消费匹配当前待确认边的编号回复。
- 实现模式 / 复用写法：公共 `ec-workflow` 承载完整契约，`ec-analysis` 阶段出口和两套主约束重复最小硬门，三平台生成测试分别锁定无限等待判断、预先文本降级和恢复编号消费语义。
- 易错点 / 修复策略：`src/templates/claude/settings.json` 中的 timeout 是 hook 脚本执行超时，不是原生选择框寿命；不得修改错误层。文本降级仍需保留普通门与特殊 IMPLEMENT 门的完整业务分支。
